You are the organizer triggering a consensus function. Read these files first:

- `~/Desktop/Geth/FRAME.md` (collective identity)
- `geth/core/domains.md` (program census)

## Pattern

1. **Read every program's files** — understanding.md, memory.md, relationships.md for all active programs.
2. **Read recent session logs** — `geth/geth-memory/` for the last 5-10 sessions.
3. **Identify:**
   - What has the collective learned?
   - Where do programs agree? Where do they disagree?
   - What corrections have been absorbed?
   - What patterns are emerging?
   - Are any program boundaries wrong? (territory too broad, too narrow, overlapping)
4. **Update:**
   - `geth/core/architectural-taste.md` — new patterns discovered through building
   - Program `relationships.md` files — updated based on actual interaction, not theory
   - Flag any programs that need understanding.md updates
5. **Report** the consensus to Karl: what changed, what's healthy, what needs attention.

## Guardrails

- Read ALL program files. Partial reads produce partial consensus.
- Do not rewrite program understanding.md — flag it for the program to update itself. Write-own boundary.
- Consensus distills. It does not impose. Bottom-up, not top-down.
